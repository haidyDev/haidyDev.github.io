Before/After: Booking System Architecture

Architecture thinking demo: current-state pain points and a proposed Power Platform target design.

← Back to Demos Projects

Before (current state)

  • Bookings are managed in multiple spreadsheets and email chains.
  • Status tracking is inconsistent, so teams duplicate work.
  • No single source of truth for customer, reservation, and payment data.
  • Manual handoffs cause delays and missed updates for end users.

After (proposed solution)

  • Model booking data in Dataverse with clear ownership and validation rules.
  • Use a Power App for booking intake, updates, and status visibility.
  • Automate notifications and approvals with Power Automate flows.
  • Provide role-based dashboards for operations and customer service.

Key decisions

  • Prioritize a normalized data model before UI complexity.
  • Separate process automation from UI logic for maintainability.
  • Define a minimal first release focused on core booking lifecycle.
  • Use environment strategy (dev/test/prod) from the start.

Risks & mitigations

  • Risk: Scope creep. Mitigation: strict MVP boundaries and backlog triage.
  • Risk: Data quality issues. Mitigation: validation rules and import cleanup flow.
  • Risk: Low adoption. Mitigation: role-based training and pilot rollout.
  • Risk: Flow failures. Mitigation: alerting, retries, and support runbooks.
